Internal proposal · Observability

Wrap Langfuse once. Reuse it everywhere.

A thin Python library that sits between our projects and Langfuse. Teams instrument with one-line helpers instead of learning the Langfuse SDK — and when Langfuse changes, one library changes, not every project.

Built & working 78 tests, ruff + mypy clean CI green · py 3.10–3.12 Live-verified vs real Langfuse

The idea

One insulating layer between our code and the vendor

Every project talks to langfuse_wrapper. Only the wrapper talks to Langfuse. That single seam is where cost tracking, PII policy, and SDK details live — defined once, inherited by all.

The difference

Same instrumentation, half the surface area

The demo integrated Langfuse straight into the app. Here is that same work — recording a trace, an LLM call, usage, and a score — written both ways.

        resp = client.messages.create(...)

  • SDK internals leak into application code
  • Repeated — and drifting — in every project
  • Langfuse's v3→v4 rewrite would break all of them
  • No shared cost, PII, or naming policy
ProposedOne wrapper, imported
import langfuse_wrapper as lw

@lw.trace(name="research_agent")
def run_agent(query):
    with lw.generation("llm-call",
                       model="claude-opus-4-8") as gen:
        resp = client.messages.create(...)
        lw.track_llm(gen, resp,
                     model="claude-opus-4-8")  # usage + cost
    lw.score("relevance", 0.9)

with lw.trace_context(session_id="s1",
                     user_id="u1", tags=["prod"]):
    run_agent("...")
  • One import, no SDK internals in app code
  • Cost, PII scrubbing, and degradation handled once
  • SDK swaps happen inside the wrapper, not the app
  • No keys? Every call is a safe no-op — runs in dev/CI

Why it pays off

Four things we get for the price of one library

Change once

Insulation from the vendor

Langfuse already shipped a breaking SDK rewrite (v3→v4). With direct integration, every project rewrites. With the wrapper, only the wrapper does — the same shield if we ever change observability vendors.

Consistency

Policy defined in one place

Cost tracking, PII scrubbing, trace naming, and session/user tagging live in the wrapper and apply everywhere — instead of each team re-implementing them and each getting it subtly wrong.

Velocity

A lower barrier for teams

A developer writes @lw.trace, not OpenTelemetry spans. Onboarding a new project to full observability takes minutes, not a spike.

Safety

Graceful by default

Missing keys or Langfuse down? Every call becomes a no-op and the app runs unchanged — no if enabled: branches scattered through product code.

The adoption path

Projects import langfuse_wrapper — never langfuse directly.

That one convention gives us a single client per process, one shared policy, and none of the cross-project drift that makes observability rot over time.

  1. pip install the wrapper from our internal package source.
  2. Set LANGFUSE_* env vars per environment; leave them unset in dev/CI.
  3. Replace direct Langfuse calls with lw.trace / generation / track_llm / score.
